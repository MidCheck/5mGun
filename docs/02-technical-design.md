# 02 · 技术方案与实施文档

> 版本 v0.1 · 状态：待评审
> 技术栈：**Three.js + TypeScript（客户端）/ Node.js + Colyseus（权威服务器）/ WebSocket**

---

## 1. 架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                         客户端 (浏览器)                         │
│  ┌──────────┐  ┌───────────┐  ┌──────────┐  ┌─────────────┐  │
│  │ 渲染层    │  │ 输入层     │  │ 预测层    │  │ 音频层       │  │
│  │ Three.js │  │ KB/触屏    │  │ 本地预测   │  │ WebAudio    │  │
│  └────┬─────┘  └─────┬─────┘  └────┬─────┘  └──────┬──────┘  │
│       └──────────────┴─────────────┴───────────────┘          │
│                     游戏状态 (本地副本)                          │
└──────────────────────────────┬──────────────────────────────┘
                                │  WebSocket (二进制/JSON)
                     输入命令 ↑      ↓ 状态快照
┌──────────────────────────────┴──────────────────────────────┐
│                   权威服务器 (Node.js + Colyseus)              │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────────┐  │
│  │ 匹配/大厅服务  │  │ 房间 (Room)   │  │ 持久化              │  │
│  │ Matchmaking  │→ │ 模拟循环 tick  │← │ SQLite→PostgreSQL  │  │
│  │              │  │ 命中/物理/AI   │  │ 账号/成长/经济      │  │
│  └──────────────┘  └──────────────┘  └────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

**核心原则：服务器权威（Server-Authoritative）。** 客户端只是"输入采集 + 渲染 + 预测"，所有命中、伤害、金币、丧尸 AI 都在服务器上裁决。这是 FPS 反作弊与防不同步的底线。

## 2. 技术栈选型

| 关注点 | 选型 | 理由 |
|---|---|---|
| 渲染 | **Three.js** | 成熟、文档全、纯 Web、可控性强 |
| 构建 | **Vite + TypeScript** | 快、HMR、类型安全 |
| 房间/同步 | **Colyseus** | Node 生态房间框架，自带状态同步(schema)、房间生命周期、客户端 SDK |
| 传输 | **WebSocket**（Colyseus 默认） | 可靠有序，门槛低；后续可评估 WebRTC DataChannel 降延迟 |
| 物理/碰撞 | 服务器轻量自研（胶囊体 + 射线），客户端 **可选** Rapier(wasm) 做表现 | FPS 多用 hitscan，不需重物理引擎；保持服务器可控、低开销 |
| 音频 | **WebAudio API** + PannerNode | 空间化、低延迟 |
| 动画 | Three.js AnimationMixer / 骨骼 + Mixamo 动画 | |
| 状态/UI | 轻量（原生 + 少量 React 仅用于菜单/HUD，游戏内 HUD 用 Canvas/DOM） | 游戏循环不进 React |
| 持久化 | **SQLite**（原型）→ **PostgreSQL**（上线） | 先轻后重 |
| 缓存/会话 | Redis（多服扩展期引入，做房间发现/匹配队列） | |

## 3. 网络模型（FPS 核心难点）

目标：在 50–150ms 抖动网络下，**自己手感跟手、别人移动平滑、命中公平**。采用经典三件套：

### 3.1 客户端预测（Client-Side Prediction）

- 玩家输入立即在本地应用（移动/开火表现），不等服务器往返 → **跟手**。
- 每个输入帧带 `seq` 序号发给服务器。

### 3.2 服务器校正（Reconciliation）

- 服务器按权威模拟，回发"已处理到 seq=N 时你的权威状态"。
- 客户端用该权威状态**回滚并重放**(replay) seq>N 的未确认输入，纠正偏差。
- 偏差小时平滑插值，避免拉扯感。

### 3.3 实体插值（Entity Interpolation）

- 其他玩家/丧尸的状态在客户端**延迟约 100ms 插值渲染**（在两个快照间插值）→ **别人移动平滑**。

### 3.4 滞后补偿（Lag Compensation，仅 PvP 命中）

- 玩家开火时，服务器**回溯**到"该客户端当时看到的世界状态"（按其 RTT/插值延迟），在历史快照上做命中判定 → **所见即所中**。
- 保留每名玩家近 ~1s 的位置历史环形缓冲用于回溯。

### 3.5 频率与带宽预算

| 参数 | 取值 | 说明 |
|---|---|---|
| 服务器 tick | **20–30 Hz** | 模拟与命中裁决频率 |
| 快照下行 | 20 Hz（可降频 + 增量） | 仅发可见/变化实体（兴趣管理） |
| 输入上行 | 30–60 Hz（合并发送） | 带 seq |
| 渲染 | 60 FPS（与网络解耦） | |
| 单玩家带宽目标 | < 30–50 KB/s | 二进制编码 + 增量 + AOI 裁剪 |

> **兴趣管理（AOI）**：只给客户端同步其视野/范围内的实体，PvE 大量丧尸时尤其关键。

## 4. 命中判定

- **武器以 hitscan（射线）为主**（AR/SMG/SG/SR/手枪），子弹视觉是表现层 tracer，判定走射线。
- 服务器对目标用**分段碰撞体**：头（爆头判定）、躯干、四肢，各自伤害倍率。
- PvP：射线判定在**滞后补偿回溯的历史快照**上做。
- PvE：丧尸由服务器模拟，判定用当前权威状态。
- 投掷物/特殊弹（手雷等，后期）走简化弹道模拟。
- **客户端只显示"疑似命中"反馈（hitmark）但不决定伤害**，最终以服务器结算为准（轻微的"反悔"用平滑处理）。

## 5. Colyseus 房间设计

### 5.1 房间类型

| Room | 职责 |
|---|---|
| `LobbyRoom` | 大厅/房间列表、创建、邀请码加入、人数填充 |
| `TdmRoom` | PvP 团队歼灭对局 |
| `ZombieRoom` | PvE 丧尸合作对局 |

### 5.2 同步状态 Schema（Colyseus @colyseus/schema）

```
RoomState
 ├─ players: Map<sessionId, Player>     // pos, rot, hp, armor, weapon, ammo, team, kills...
 ├─ projectiles: []                     // 表现用（如有非 hitscan）
 ├─ zombies: Map<id, Zombie>            // PvE: type, hp, pos, target, state
 ├─ phase: enum                         // warmup / playing / intermission / boss / ended
 ├─ scores / waveIndex / timer
 └─ economy (PvE 局内金币等)
```

### 5.3 消息（client → server）

`input`(移动/视角/动作 + seq)、`fire`(方向/时间戳)、`reload`、`switchWeapon`、`buyUpgrade`、`revive`、`chat`。

### 5.4 房间生命周期

`onCreate` 载入地图配置 → `onJoin` 鉴权+落座+发初始快照 → 固定步长 `setSimulationInterval`(tick) 跑模拟 → `onLeave` 处理掉线（短重连窗口）→ 结算 `onDispose` 写库。

## 6. 服务器模拟循环（每 tick）

```
1. 收集本 tick 各玩家输入（真人按 seq 排序去抖；PvP 机器人在服务器内生成等价输入）
2. 应用移动（胶囊体 + 地图碰撞，含防穿墙/速度上限校验）
3. 处理开火 → 滞后补偿回溯 → 射线命中 → 扣血/记伤/判定爆头
4. PvE: 丧尸 AI（寻路/索敌/攻击）+ 波次生成器（按真人数缩放）+ 掉落
5. 结算死亡/复活/金币/技能效果
6. 生成增量快照（按 AOI 裁剪）→ 下发各客户端
7. 反作弊校验（速度/射速/命中率异常打点）
```

> 丧尸寻路：网格/导航网格(NavMesh) 简化版 + 流场（flow field）应对成群涌动，避免每只独立 A* 的开销。

### 6.1 PvP 机器人（Bot Fill，服务器侧）

- 机器人是**纯服务器实体**，复用真人同一套移动/开火/命中管线（无需信任客户端，天然防作弊）。
- 简化 AI 状态机：`巡逻 → 索敌 → 寻掩体 → 对枪 → 撤退/换弹`；难度档位调反应时延、瞄准噪声、走位质量、失误率（拟人化，不做神枪手）。
- 补位逻辑：`onJoin/onLeave` 时按目标人数动态增删机器人，保持两队平衡；真人进入优先顶替机器人。
- 对客户端而言机器人与真人同构（同一 Player schema + `isBot` 标记），渲染/计分一致。

### 6.2 PvE 动态难度缩放（按真人数量）

波次生成器读取**当前真人数 `n`** 与 `shared/config` 里的缩放系数，实时决定每波的怪量与强度：

```
countThisWave   = base[wave] * countMul[n]          // 怪量随人数放大
zombie.hp       = base.hp     * hpMul[n]            // 血量小幅上调
zombie.damage   = base.damage * dmgMul[n]
eliteRatio      = base.elite  + eliteBonus[n]       // 高级怪占比随人数升
boss.hp/skills  = base * bossMul[n]                 // BOSS 分档放大
coinDrop        = base * coinMul[n]                 // 同步微调，保人均成长一致
```

- 系数表 `countMul/hpMul/...` 全部进 `shared/config`，**热调不改代码**。
- **中途进出平滑**：玩家加入/掉线只影响**下一波**的生成参数，当前波不突变，避免过载或空场。
- **同屏实体上限**仍受性能预算约束（§11）：单人时主动下调同屏怪量与近身围殴密度，给走位空间；人多时优先增加"成波涌入"而非无限堆叠同屏数。
- 单人保留限次自救（GDD §2.2/§2.4）的服务器端实现：倒地进入可救援计时，单人触发自救而非等队友。

## 7. 防作弊

| 风险 | 对策 |
|---|---|
| 自瞄/透视 | **服务器权威**，客户端不掌握判定；命中率/爆头率异常监控 |
| 加速/瞬移 | 服务器校验位移上限，超阈值钳制并打点 |
| 加速开火 | 服务器侧射速节流（按武器冷却） |
| 伪造伤害 | 伤害只由服务器计算，客户端无权写 |
| 资源/金币篡改 | 经济全在服务器，客户端只读 |
| 协议滥用/刷包 | 消息频率限流、schema 校验、异常断连 |

> 客户端代码可被逆向是既定前提——**任何重要逻辑都不信任客户端**。

## 8. 数据模型与持久化

```
Account     id, nickname, guest?, createdAt, lastLogin
Profile     accountId, level, exp, softCurrency, hardCurrency
Unlocks     accountId, weaponId/skinId/...（永久解锁）
Loadout     accountId, 选中的武器/皮肤/技能配置
MatchLog    matchId, mode, players[], 战绩, 时间（用于结算与风控）
```

- 原型期：SQLite + 简单 ORM（Drizzle/Prisma）。
- 上线期：PostgreSQL；高频会话/匹配队列用 Redis。
- **PvP 永久数据不含战斗数值优势**（仅外观/便利），保证竞技公平（见 GDD §7.3）。

## 9. 客户端架构

```
src/
  core/        游戏循环、时钟、固定步长累加器
  net/         Colyseus client、预测、校正、插值、快照缓冲
  render/      Three.js 场景、相机、灯光、后处理(爆头慢镜/震屏)
  input/       PC(KB+PointerLock) / Mobile(虚拟摇杆) 统一成 InputCommand
  entities/    本地玩家、远端玩家、丧尸、武器、子弹 tracer
  weapons/     武器表现、后坐、换弹动画、枪口火光
  audio/       WebAudio 管理、空间化、连杀播报
  fx/          爆头特效、命中粒子、HitMarker、伤害飘字
  ui/          菜单/房间/HUD/结算
  config/      武器/丧尸/数值 JSON（与服务器共享同一份）
```

- **客户端与服务器共享数值配置与碰撞参数**（同一 `config/` 与 `shared/` 包），保证预测与权威一致。
- 渲染与网络**解耦**：渲染按 requestAnimationFrame，网络按快照；用插值桥接。

## 10. 移动端适配

- 触屏检测 → 加载虚拟摇杆 UI（nipplejs 或自研）。
- 自适应分辨率/像素比，低端机降阴影、降粒子、降丧尸可见数。
- 性能预算：中端手机 **稳定 30+ FPS**；提供画质档位（高/中/低）。
- 可选辅助瞄准（aim assist）降低触屏门槛，仅 PvE 或 PvP 轻度。

## 11. 性能预算

| 项 | 目标 |
|---|---|
| 首屏可玩加载 | < 5s（资源分包、按需加载地图/武器） |
| 客户端帧率 | PC 60 / 移动 30+ |
| 同屏丧尸（PvE） | 服务器实体上限 ~40–60，客户端用实例化(InstancedMesh)渲染 |
| 单房承载 | **PvP 上限 20 人 / PvE 上限 8 人**，对局内 tick < 几 ms |
| 包体 | 核心 JS gzip < 1–2 MB，素材懒加载 |

优化手段：InstancedMesh（丧尸/子弹）、纹理图集、几何合并、对象池、LOD、增量快照、二进制协议。

> **20 人房的网络与性能注意**：玩家两两可见时同步量约 O(n²) 增长，20 人是首期最大压力点——**AOI 兴趣管理（§3.5）+ 增量/二进制快照 + 快照降频**是必须项，不能等优化期再补。20 人房用更大的地图/出生分布避免挤成一团。机器人 AI 也要控制单 tick 预算（分帧/简化），20 人满机器人时不拖垮 tick。

## 12. 部署与扩展

### 12.1 起步（单云主机）

- 1 台云主机：Node 进程跑 Colyseus（大厅 + 若干房间）+ SQLite + Nginx 反代/TLS + 静态资源(CDN)。
- 满足验证期数十到数百并发。

### 12.2 扩展（弹性）

- Colyseus 多进程/多节点 + Redis 做 presence/房间发现 + 匹配服务。
- 按区域部署降低延迟；房间编排按负载起停进程。
- 静态资源走 CDN，WebSocket 走就近边缘。

### 12.3 可观测

延迟/丢包/tick 耗时/房间数/异常断连/作弊打点 → 指标面板与告警。

## 13. 安全与合规

- HTTPS/WSS 全程加密；昵称/聊天做敏感词过滤。
- 游客可玩 + 可选轻量账号；隐私数据最小化收集。
- **素材合规**：仅用 CC0/CC-BY/原创素材，建立来源台账（见 [04 素材方案](04-assets.md)）。

## 14. 关键技术风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 网络同步手感差 | 直接毁体验 | 预测+校正+插值+滞后补偿，早做 M2 打通并实测 |
| 移动端性能 | 用户面广 | 实例化渲染 + 画质档位 + AOI，早测低端机 |
| 丧尸 AI/寻路开销 | PvE 卡顿 | 流场寻路 + 实体上限 + 分帧 |
| 作弊 | 竞技崩坏 | 服务器权威从第一天就坚持 |
| 素材版权 | 法务风险 | 全程 CC/原创 + 台账 |
| 局时失控 | 偏离定位 | 硬性计时 + 节奏数值化可调 |
